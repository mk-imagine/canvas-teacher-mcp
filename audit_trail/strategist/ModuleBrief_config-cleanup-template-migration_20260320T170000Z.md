## Supplementary Analysis

### Migration Details for `create_item` (find.ts)

The `create_item` tool has three config-dependent code paths that need migration:

1. **Assignment description from `notebook_url`** (lines 685-691): Currently calls `renderTemplate(config.assignmentDescriptionTemplate.default, {...})`. Migration approach: Create `assignment-description.hbs` and `solution-description.hbs` in the defaults template directories. Use `TemplateService.renderFile()` to render them. The `registerFindTools` function already receives a `TemplateService` parameter (added in Module 2). The body files should live in a shared location (e.g., a `_shared/` template directory or within each template directory). **Decision for Tactician:** Since these descriptions are not template-specific (they apply to any `create_item` call, not just `build_module`), they may be better placed as hardcoded constants at the top of `find.ts` using the existing local `renderTemplate` Handlebars helper (line 62), avoiding coupling `create_item` to `TemplateService` for a simple two-line HTML template.

2. **Exit card quiz** (lines 725-774): Currently reads `config.exitCardTemplate` for title, quizType, questions, and `config.defaults.exitCardPoints` for points. Migration: Hardcode these as constants in `find.ts`. The exit card specification is stable and small (title pattern, graded_survey type, 3 questions, 0.5 points).

3. **Default `pointsPossible` and `submissionType`** (lines 697-710): Currently reads `config.defaults.pointsPossible` (100) and `config.defaults.submissionType` ('online_url'). Migration: Define as constants at the top of `find.ts`.

### `executeRenderables` config parameter

After migration, `executeRenderables` may no longer need `config` at all -- all current config reads are from `defaults` or `exitCardTemplate`. The Tactician should verify whether the `config` parameter can be removed entirely from `executeRenderables`'s signature, or whether it is needed for any other purpose not visible in the lines read during scoping.

### Risk: User-directory config files

The brief includes cleaning `~/.config/mcp/canvas-mcp/config.json` and `config.sfsu.json`. These are live files on the user's machine. The Tactician should ensure the implementation reads and rewrites these carefully (JSON parse, delete keys, JSON stringify with formatting) rather than blind replacement.
